# adapted from https://github.com/udacity/ud120-projects/issues/181

def dos2unix(path):
    """"
    convert dos linefeeds (crlf) to unix (lf)
    usage: dos2unix.py
    """

    original = path
    root = path.split('.')[0]
    destination = root +'_windows.pkl'

    content = ''
    outsize = 0
    with open(original, 'rb') as infile:
        content = infile.read()
    with open(destination, 'wb') as output:
        for line in content.splitlines():
            outsize += len(line) + 1
            output.write(line + str.encode('\n'))

def check_outliers(data, features):
    '''
    Function to evaluate possibles outliers on a dataframe
    
    input: data - dataframe with the values
           features - list of features that will be evaluated
           
    output: check_min - list of features to check their lower values
            check_min - list of features to check their upper values
    '''
    check_min = []
    check_max = []
    for feature in features:
        min_outlier_boundary = data[feature].mean() - data[feature].std()
        max_outlier_boundary = data[feature].mean() + data[feature].std()
    
        if data[feature].min() < min_outlier_boundary:
            print(f'{feature} MINIMUM value worth to check')
            check_min.append(feature)
            
        if data[feature].max() > max_outlier_boundary:
            print(f'{feature} MAXIMUM value worth to check')
            check_max.append(feature)
            
    return [check_min, check_max]

def get_outliers(condition, order='ascending'):
    '''
    Funtion to retunrs the data that meets the defined condition 
    
    input:
        condition - string to define the condition to be met
        order - The sort order
            'ascending' or 'descending'
            
    output:
        result - series with the data that meets the condition
        
    '''
    feature = condition.split()[0]
    
    if order == 'descending':
        result = df_enron.query(condition)[feature].sort_values(ascending=False)
    else:
        result = df_enron.query(condition)[feature].sort_values()
    
    print(f'From {result.shape[0]} outliers:', end='\n')
    poi, non_poi = identify_poi(condition)
    
    if len(poi) > 0:
        for person in poi:
            print(f'{person} is POI')
    
    if len(non_poi) > 0:
        for person in non_poi:
            print(f'{person} is NOT a POI')
    
def identify_poi(condition):
    '''
    Funtion to identify the POI on a subset of the dataframe defined based on the defined condition 
    
    input:
        data - dataframe
        condition - string to define the row that will be analysed
    
    output:
        poi - list with POI identified
        non_poi - list with non_POI
    '''
    poi = []
    non_poi = []
    for person in df_enron.query(condition).index:
        if df_enron.loc[person].poi:
            poi.append(person)
        else:
            non_poi.append(person)
    
    return [poi, non_poi]

2. What features did you end up using in your POI identifier, and what selection process did you use to pick them? Did you have to do any scaling? Why or why not? As part of the assignment, you should attempt to engineer your own feature that does not come ready-made in the dataset -- explain what feature you tried to make, and the rationale behind it. (You do not necessarily have to use it in the final analysis, only engineer and test it.) [relevant rubric items: “create new features”, “intelligently select features”, “properly scale features”]¶

R. The feature selection process chosen was the SelectKbest() function from sklearn. It returns the scores for the "k" best features to identify POI, where k is a parameter. The k parameter chosen was k='all', however just for two didactic reasons: analyze how each feature performs when try to identify POI and to evaluate the differences when the new features is added to the data.

Prior to any selection, it was required to perform scaling of the features due to high differences on the values. Total Payment feature goes up to hundreds of million and the e-mail messages goes only to a few hundreds. To fairly use them to evaluate importance based on the variation, it is required to put them under comparable size values. MinMaxScaler() function was used.

At the end, the 10 features with higher scores were selected to start the POI identifier. This selection took into account that the dimensionalities of the data would be reduced in 50%, however still getting many of the data variance. This number of features will be revaluated when actually creating the classifier. Based on the metrics, it will be decided if there is a chance to reduce more or if it is necessary to increase the number of features.

The features evaluated does not include the ones that had almost every value as null, because it was consider that we will be fitting the data to noise. Beside the feature removal, it was added five new features:

  1. from_poi_ratio: fraction of all e-mail messages received by a person that was sent from a POI
  2. to_poi_ratio: fraction of all e-mail messages sent by a person that was sent to a POI
  3. shared_poi_ratio: fraction of all e-mail messages received by a person that was also sent from a POI
  4. bonus_ratio: fraction of the bonus received compared to the salary
  5. schemes_citation: number of California schemes that was found on the e-mails text

The first two features were suggested by Katie on the class. The reason behind is that, if a given person, proportionally, send or receive more e-mails from a POI, this person has better chance of being a POI too.

The last three features were made by the student. The third one follows the same approach from the previous two: if a person, proportionally, receives many e-mails that a POI also receives, this person may also be a POI.

The fourth feature engineered considered that, prior to the scandal reveal, the POI distributed large amount of bonus [ 3 ] to be cashed in stock to increase their gain while the stock values were still high. The rationale is that the feature would be better predictor than salary and bonus because the higher the bonus-salary ratio is, it would be more likely that the person is a POI.

The fifth feature was built aiming to identify citation for Enron's energy schemes in California [ 6 ] to increase their profit, or, actually, reduce their losses. The e-mails on the dataset were scanned and citation for "fat boy", "megawatt laundering", "red congo" and other schemes was counted and the value associated to the e-mail address that sent the message.

Not all the features showed good results in better predict the POI. Only to_poi_ratio and shared_poi_ratio was included in the final model as a predictor and the F2 score went up by 4%.

3. What algorithm did you end up using? What other one(s) did you try? How did model performance differ between algorithms? [relevant rubric item: “pick an algorithm”]¶

R. To decide which algorithm to predict POI was better, the performance for each one tried was evaluated with tester.py file from the git cloned. Although to use the code with the sklearn version installed, is was necessary minor changes [ 7 ] to iterate though the indices of training and test set from StratifiedShuffleSplit function.

The decision for the algorithm was based on precision and recall scores for each model, with more emphasis in the recall metric. The main goal was catch the POI, even if this would misclassify some Non-POI.

**It was tried Nave Bayes, SVM, K-nearest neighbors and Adaboost algorithm and, surprisingly, the model with best performance was the simplest one tried: Nave Bayes.

**The best SVM, which initially was consider that would have the better performance, was a polynomial of third degree, C = 10 and gamma = 0.6.

K-nearest neighbors - n_neighbors=5 - and Adaboost - learning_rate=3 and n_estimator=20 - was the others best model tried. respectively.

The algorithm that most correctly classify POI (precision score) is the Nave Bayes and the best one to identify POI (recall score) is the Adaboost. The last one, however, it is recalling the POI with a trade-off from misclassification. Nave Bayes has the balance between precision and recall.

4. What does it mean to tune the parameters of an algorithm, and what can happen if you don’t do this well? How did you tune the parameters of your particular algorithm? What parameters did you tune? (Some algorithms do not have parameters that you need to tune -- if this is the case for the one you picked, identify and briefly explain how you would have done it for the model that was not your final choice or a different model that does utilize parameter tuning, e.g. a decision tree classifier). [relevant rubric items: “discuss parameter tuning”, “tune the algorithm”]¶

R. Tune the parameters means build the boundary with a shape that best classify the training points without lose the capacity of be general enough to predict the labels for test points. If the parameters is not tuned well, the model might end up having bias or variance errors. High bias is when the parameters define an under fitted model (high error for training and test data) and high variance is an over fitted model (very low error for training data, but high error for test data).

For the algorithm selected, since it was the Nave Bayes, it was not required to tune parameters. However, it was tried the Support Vector Classification (SVC) and tested for different kernel trick (polynomial, rbf), different values of C, which defines how well the boundary will fit the data (high C, high fit), gamma, defines how the data will influence the boundary (high gamma, high influence of the data, high fit). For the polynomial, it was also tested different degrees, from one, which is a linear model, and 7, which is a high degree and, probably overfitted.

These parameters were tuned applying GridSeachCV(). This function receives the model parameters as its own parameters and does a grid search with all the combinations possible fitting the training data and compute the validation score predicting the test data. The training and test split is done internally by the function.

5. What is validation, and what’s a classic mistake you can make if you do it wrong? How did you validate your analysis? [relevant rubric items: “discuss validation”, “validation strategy”]¶

R. Validation is the phase where we actually test the performance of the model in predict new data. Once the classifier is made, it has to be fitted to the data, however since it is never known the classification of new data, it wouldn't be possible to evaluate if the data was fitted properly. Therefore, after fit the classifier, it is validated on a new, but known, set of data.

One way to perform validation is splitting the data available into two groups called training and test. The training set is the one to use to fit the model or, in order words, let the model learn the patterns. The test group is used to predict a value using the features and compare it with the actual value. This comparison results on validation scores - such as precision and recall that is used to evaluated the model.

The most common error made on the validation process is the usage of the test data to perform training of the model. The effect of the error is the tendency of the model be overfitted, since it already learn the pattern of the test data.

On this project, it was not built a different validation process different that the one in tester.py. It was tried, however, since we have few POI, when the training and test data were split, sometimes one of the groups end ups not having both class of labels to run properly.

The algorithm on tester.py avoid this scenario with StratifiedShuffleSplit() function. By using it, the function assure that both sets will have the same proportion of class labels. Furthermore, this function also allows to take advantage of the k-fold validation, when the data is splited between training and test several times, always fitting and validating the model. It is very helpful when a small dataset is used since the model is trained and test in all the data.

6. Give at least 2 evaluation metrics and your average performance for each of them. Explain an interpretation of your metrics that says something human-understandable about your algorithm’s performance. [relevant rubric item: “usage of evaluation metrics”]¶

R. It was evaluated two metrics for this project: precision and recall.

The precision is the ability of the model to correctly classify the points. On the data evaluated, the precision considers all the points that was classified as POI and evaluates how many was actually a POI. It is a metric that evaluate how many times the model hit the target. At average, all the classifiers tried made a fair good job in correctly classify as POI, the precision vary from 0.13 until 0.44.

The ability of recognize the classification of the point is called the recall. Bringing this definition to enron data, it is the ability of, given that a person is a POI, the model would label him as a POI. It is a metric that evaluate how well the model is resembling the patterns learned from the training point.

The models tried had very different performance on this metrics: from 0.06 until 0.97. Both extremes are not good: one is not recalling any POI and the other is identifying all the POI, however too good to be true. It is probably considering every point as POI, therefore not losing any, however trading-off with misclassification. If this is an acceptable approach, machine learning is not required.
